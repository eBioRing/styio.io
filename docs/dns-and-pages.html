DNS and Pages

Start with one GitHub Pages site and keep the client-facing base URL stable enough to migrate behind a CDN later.

Startup setup

  • Configure this repository's Pages site with the custom domain styio.io.
  • Use GitHub Releases as the durable source for release-root bundles.
  • Use the Pages workflow to publish docs plus the compatible static read root.

For GitHub Actions based Pages deployments, the custom domain is a repository Pages setting. The checked-in CNAME file is kept as a visible domain marker, but the Pages setting is the source of truth.

Remove registrar parking, URL forwarding, and wildcard records before expecting GitHub Pages to serve the domain. HTTPS enforcement can be enabled after GitHub issues the certificate.

Package domain

During alpha, the public install command can use https://styio.io as the release root. If a dedicated packages.styio.io subdomain is introduced, point it at an equivalent static read plane and keep the same path contract. 

curl -fsSL https://packages.styio.io/tools/spio/install-spio.sh | sh -s -- --base-url https://packages.styio.io

Migration path

The static layout keeps the migration surface small. A future CDN, object store, or self-hosted platform service only needs to serve the same tools/ paths or provide a client-compatible redirect layer after spio learns that protocol.
